The ZEOS Book

ZEOS Action Circuit

As in Zcash Orchard there is only one circuit which is used to generate proofs for all privacy actions. The ZEOS Orchard circuit, Czeos​, is very similar to the Zcash Orchard circuit. It can be divided into three parts: A, B and C. Each circuit part represents a UTXO and the action circuit describes their relationship to each other. There are two main configurations for this circuit.

It is either:

  1. A=B+C

or

  1. A=C=0

The first configuration is used for all transfer and burn actions. UTXO A represents the note which is being spent by the action. UTXO B represents the receiving part of the action whereas UTXO C represents the 'change' which goes usually back into the wallet of the sender (spender of UTXO A). Hence the relation A=B+C between the UTXOs.

In case of NFT transfers (or burns) UTXO C is always zero which enforces A=B. This statement must be true for NFT transfers since NFTs are not divisable.

The second configuration is used for minting notes only. The configuration A=B=0 effectively disables the circuit parts A and C leaving only part B enabled.

Private Inputs (ω)

The following list contains all private inputs to the top level ZEOS action circuit.

Note A (action input):

  1. path : Authentication path of note commitment A. The sister path of note commitment A which is required in order to calculate it's merkle plath.
  2. pos : Position of note commitment A inside the merkle tree. Specifically this is the leaf index of note commitment A.
  3. gd​a​ : Address diversify hash of note A. Deterministically derived from a diversifier index (see p. 37 Zcash Protocol Specification).
  4. pkd​a​ : Address public key of note A. Derived from Incoming Viewing Key and diversify hash (see p. 14 and 37 Zcash Protocol Specification).
  5. d1a​ : Either the value of note A (fungible token) or the (lower 64 bits of the) unique identifier of note A (non-fungible token).
  6. d2a​ : Either the symbol code of note A (fungible token) or the (upper 64 bits of the) unique identifier of note A (non-fungible token).
  7. ρa​ : Randomness to derive nullifier of note A (equals nullifier of note that was spent in order to create note A) (see p. 14 Zcash Protocol Specification)
  8. ψa​ : Additional randomness to derive nullifier of note A (see p. 14 Zcash Protocol Specification)
  9. rcma​ : Random commitment trapdoor of note commitment A (see p. 14 and 28 Zcash Protocol Specification)
  10. cma​ : Note commit of note A (see p. 28 Zcash Protocol Specification).
  11. α : Randomness to derive a spend authorization signature for note A (see p. 55 Zcash Protocol Specification).
  12. ak : Spend Validating Key which is part of the Full Viewing Key components (see p. 36 and 116 Zcash Protocol Specification).
  13. nk : Nullifier Deriving Key which is part of the Full Viewing Key components (see p. 36 and 116 Zcash Protocol Specification).
  14. rivk : Randomness which is part of the Full Viewing Key components to derive corresponding Incoming Viewing Key of the address diversify hash of note A (see p. 36, 37 and 116 Zcash Protocol Specification).

Note B (action output):

  1. gd​b​ : Address diversify hash of note B.
  2. pkd​b​ : Address public key of note B.
  3. d1b​ : Either the value of note B (fungible token) or the (lower 64 bits of the) unique identifier of note B (non-fungible token).
  4. d2b​ : Either the symbol code of note B (fungible token) or the (upper 64 bits of the) unique identifier of note B (non-fungible token).
  5. scb​ : The code of the smart contract issuing notes A, B and C.
  6. ρb​ : Randomness to derive nullifier of note B (equals nullifier of note A).
  7. ψb​ : Additional randomness to derive nullifier of note B.
  8. rcmb​ : Random commitment trapdoor of note commitment B.
  9. accb​ : Code of EOS account in which this note is 'burned'.

Note C (action output):

  1. gd​c​ : Address diversify hash of note C.
  2. pkd​c​ : Address public key of note C.
  3. d1c​ : Value of note C (fungible token only).
  4. ψc​ : Additional randomness to derive nullifier of note C.
  5. rcmc​ : Random commitment trapdoor of note commitment C.
  6. accc​ : Code of EOS account in which this note is 'burned'.

Public Inputs (x)

The following list contains all public inputs to the top level ZEOS action circuit.

  1. ANCHOR : Merkle tree root of authentication path of note A (see p. 17 to 19 Zcash Protocol Specification).
  2. NF : Nullifier of note A (see p. 56 Zcash Protocol Specification).
  3. RKX​ : Spend Authority (x component) of (α, ak) (see p. 61 Zcash Protocol Specification).
  4. RKY​ : Spend Authority (y component).
  5. NFT : Indicates if notes in circuit represent NFTs or fungible tokens.
  6. Bd1​ : Exposes value (fungible token) or unique id (non-fungible token) of note B in case of MINT or BURN.
  7. Bd2​ : Exposes symbol (fungible token) or unique id (non-fungible token) of note B in case of MINT or BURN.
  8. Bsc​ : Exposes smart contract code of note B in case of MINT or BURN.
  9. Cd1​ : Exposes value of note C in case of BURNFT2 (fungible tokens only).
  10. CMB​ : Note commitment of note B.
  11. CMC​ : Note commitment of note C.
  12. ACCB​ : Code of EOS account in which note B is 'burned'.
  13. ACCC​ : Code of EOS account in which note C is 'burned'.

Circuit Internal Signals

The following signals are circuit-internal only.

  1. root : Derived root of merkle path of note A.
  2. cma′​ : Derived commitment of note A.
  3. pkd​a′​ : Derived address public key of note A.
  4. rkx​ : Derived spend authority (x component) of note A.
  5. rky​ : Derived spend authority (y component) of note A.
  6. nfa​ : Derived nullifier of note A.
  7. cmb​ : Derived commitment of note B.
  8. cmc​ : Derived commitment of note C.

Constraints

Constraining an arithmetic circuit in Halo 2 is very similar to integrated circuit design in computer engineering. All Constraints are expressed as mathematical equations evaluating to zero. The logical AND becomes a multiplication and the logical OR becomes an addition.

The following statements for private and public inputs must hold.

The global statement 'either A=B+C or A=C=0' results in the following constraint for the note values d1a​,d1b​,d1c​:

  1. (d1a​−d1b​−d1c​)⋅(d1a​+d1c​)=0

For circuit part A the following constraints must hold:

  1. d1a​⋅(root−ANCHOR)=0
  2. d1a​⋅(cma​−cma′​)=0
  3. d1a​⋅(pkd​a​−pkd​a′​)=0
  4. d1a​⋅(rkx​−RKx​)=0
  5. d1a​⋅(rky​−RKy​)=0
  6. d1a​⋅(d2a​−d2b​)=0
  7. d1a​⋅(nfa​+ρb​−2⋅NF)=0

For circuit part B the following constraints must hold:

  1. Bd1​⋅(Bd1​−d1b​)=0
  2. Bd1​⋅(Bd2​−d2b​)=0
  3. Bd1​⋅(Bsc​−scb​)=0
  4. CMB​⋅(CMB​−cmb​)=0
  5. ACCB​−accb​=0

For circuit part C the following constraints must hold:

  1. NFT⋅d1c​=0
  2. Cd1​⋅(Cd1​−d1c​)=0
  3. CMC​⋅(CMC​−cmc​)=0
  4. ACCC​−accc​=0
